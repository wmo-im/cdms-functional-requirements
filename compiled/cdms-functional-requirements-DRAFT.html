1. Scope

1.1. Introduction

1.1.1. Purpose and Relationship to WMO-No. 1131

This document serves as a companion specification to the Climate Data Management System Specifications (WMO-No. 1131, 2025 Edition), published by the World Meteorological Organization (WMO). WMO-No. 1131 provides a foundational framework for Climate Data Management Systems (CDMS), detailing essential components, governance principles, and operational guidelines using classifications of "Required," "Recommended," or "Optional."

This specification translates the framework described in WMO-No. 1131 into concrete functional and non-functional requirements. The functional and non-functional requirements form a subset of the full specification defined in WMO-No. 1131, focussing on those elements essential for the management of observational data from stations, including the management of the station, sensor and provenance metadata. The functional requirements are expressed as requirements classes following the OGC Modular Specification approach.

This specification aims to:

  • Assist Developers and Integrators: Provide practical, testable requirements and detailed technical considerations for the design, development, and integration of robust and compliant CDMS components and systems.

  • Empower Users and Assessors: Establish a comprehensive set of conformance criteria and common understanding to objectively assess the capabilities and compliance of existing, or proposed, CDMS implementations with the essential functionalities described in WMO-No. 1131.

  • Enable Interoperability: Define standardized interfaces, data formats, and protocols to facilitate integration between CDMS implementations and external systems within the climate data ecosystem.

This specification is not intended as a list of requirements that WMO Members SHALL or SHOULD follow.

1.1.2. System Context and Boundaries

Figure 1 illustrates the architectural landscape and positioning of a Climate Data Management System within the broader weather, climate, and other environmental data value chain (hereafter, climate data for conciseness). Climate data typically flows from field sensors and observing stations through national Data Collection and Processing Systems (DCPS), and subsequently to various downstream applications and services.

Key downstream systems include:

  • WIS 2.0 nodes for international data exchange

  • Operational meteorological systems (Numerical Weather Prediction, data assimilation)

  • Climate Data Management Systems for quality control, homogenization, and archival

  • Regional Climate Centres for regional climate monitoring and services

  • Long-term preservation archives and public-facing data portals

Figure 1. System landscape showing the context of Climate Data Management Systems within the data exchange architecture

Scope Boundaries:

While some operational implementations may integrate multiple components of the data flow—for example, combining DCPS and CDMS functionality within a single system—this specification focuses exclusively on the Climate Data Management System component and its specific responsibilities.

The following are explicitly OUT OF SCOPE for this specification:

  • Field observation collection (sensors, data loggers, first-mile infrastructure)

  • Real-time data collection and processing systems (DCPS)

  • Time-critical data exchange via WIS 2.0 for operational meteorology

  • Numerical Weather Prediction and forecasting systems

  • General-purpose meteorological databases not specifically designed for climate data

  • Operational quality control for real-time applications

The following are IN SCOPE:

  • Quality control procedures specific to climate data requirements with emphasis on long-term consistency

  • Detection and correction of inhomogeneities in climate time series (homogenization)

  • Climate data rescue, including digitization of historical paper records

  • Station metadata management with emphasis on long-term consistency and provenance

  • Archival and preservation of climate datasets with appropriate documentation

  • Production, publication and distribution of quality-controlled climate data products

  • Integration interfaces with WIS 2.0, national networks, and international archives

  • Data lineage tracking and provenance management

This specification recognizes that CDMS implementations may exist within national meteorological services (National Segment) and contribute to regional and international climate services (Regional/International Segments) as shown in Figure 1. Requirements are designed to support both contexts while maintaining flexibility for local operational arrangements.

1.2.2. Container Architecture (Level 2)

Figure 3 provides a high-level view of the major functional components and data stores within the Climate Data Management System. This diagram is intentionally technology-agnostic, allowing implementers flexibility in architectural and technology choices while conforming to the functional requirements of this specification.

Figure 3. Container diagram showing major functional components and data stores

Key Containers:

  • User Interface: Provides interactive capabilities for climate scientists, data managers, and system operators to review quality-controlled data, validate observations, manage station metadata, configure system parameters, and monitor system operations. This may be implemented as a web application, desktop application, or a combination of both.

  • Data API: Exposes standardized interfaces for data ingestion from external sources, data query and retrieval by users and downstream systems, and publication of climate data products. The API should support international standards for climate data exchange including OGC API standards.

  • Data Ingestion: Processes incoming observations from the national DCPS (national real-time and non-real-time data) and from WIS 2.0 (international observations), as well as digitized historical observations submitted by the Data Rescue Service. Handles multiple data formats and protocols, performing initial validation and normalization.

  • Data Processing Engine: Applies quality control algorithms to climate observations including range checks, temporal consistency checks, spatial consistency checks, and climatological validation. Also performs statistical analysis to detect and adjust for non-climatic discontinuities in observation time series (homogenization), producing both adjusted data and comprehensive documentation of the process.

  • Metadata Management: Maintains comprehensive station and element metadata including location information, instrumentation history, observing practices, element definitions (units, codes, and controlled vocabularies), and data lineage. Supports both data ingestion validation and quality control processing, and ensures compliance with international metadata standards.

  • Data Rescue Service: Manages the digitization workflow for historical climate records, including coordination of document imaging, data entry from paper records, and quality assurance of digitized observations. Preserves both the scanned images and the keyed data with full provenance information. Digitized observations flow through the same Data Processing Engine as contemporary observations to ensure consistent quality standards.

  • Data Catalogue: Maintains persistent discovery metadata records describing available datasets, observations, and metadata holdings. Supports search and discovery via the Data API and publishes discovery metadata to downstream services.

  • Archive and Backup: Manages scheduled backup and archival of the Data Store and File Storage contents. Submits datasets to external long-term preservation archives and maintains backup copies to support disaster recovery.

Data Stores:

  • Data Store: Persistent storage for climate observations with their associated quality flags and processing history, station and element metadata (including units, codes, and controlled vocabularies), quality control rules and thresholds, system configuration, homogenization parameters, and data lineage information. Maintains historical records of all metadata changes to support long-term climate analysis.

  • File Storage: Manages climate-standard file formats (such as NetCDF-CF), quality control reports, homogenization analysis outputs, digitized document images, and archived datasets. Supports versioning and long-term preservation requirements.

External System Integration:

The system integrates with external systems at well-defined conformance boundaries where standardized protocols and data formats apply:

  • WIS 2.0: International data exchange using MQTT messaging and standardized data formats for both consuming international observations and publishing national climate products.

  • National Data Collection and Processing System: Real-time and non-real-time data sources and processing, including connection to data loggers, sensors and other external data sources.

  • Long-term Preservation Archives: Climate dataset archival using standard formats.

  • Downstream Services: External consumers of climate data and metadata, including climate data stores, analysis platforms, and WIS 2.0 nodes, accessed via the Data API and Data Catalogue.

  • Identity Management Systems: Enterprise authentication and authorization using standard protocols (LDAP, OAuth 2.0, SAML).

This architectural decomposition supports the modular requirements structure of this specification, where different containers may be subject to different requirements classes based on their functional responsibilities.

1.3. Document Structure and Organization

To facilitate clear understanding and practical application, this document is structured into the following key sections:

  • Functional Requirements: Details what a CDMS shall do, providing specific, granular requirements for core functionalities such as data ingestion and validation, transformation and storage, metadata management, quality control, data rescue, processing, and user interface interactions. These requirements operationalize the high-level functional components outlined in WMO-No. 1131.

  • Non-Functional Requirements: Defines how well a CDMS shall perform, covering essential attributes like performance, scalability, reliability, maintainability, usability, interoperability, data governance, and comprehensive security measures. These requirements ensure that a CDMS not only fulfills its functional duties but also operates effectively, securely, and sustainably in various environments.

  • Appendix: Provides supplementary information, including a glossary of terms, a list of acronyms used throughout the document, references to related standards and specifications, and any other relevant supporting material that aids in the understanding and application of these specifications.

Each requirement within the functional and non-functional sections is designed to be clear, measurable, and directly applicable, serving as a valuable reference for both system development and compliance assessment activities in the context of WMO-No. 1131.

2. Conformance

This specification defines the following conformance classes corresponding to the functional capabilities of a Climate Data Management System:

  • Core Data Management (Required): Foundational observation storage and retrieval, provenance metadata management, and basic station and sensor metadata storage. This class provides the data management foundation on which all other conformance classes depend.

  • Quality Control (Required): Climate-specific QC procedures including range, temporal, spatial, and climatological checks

  • Data Ingestion (Required): Multi-format data import, validation, and normalization

  • Data API (Required): Standards-based data query, retrieval, and distribution interfaces

  • Homogenization (Recommended): Statistical detection and adjustment of inhomogeneities in climate time series

  • Data Rescue (Optional): Digitization workflow for historical paper records including imaging and data entry

  • Metadata Management (Required): Advanced station and element metadata management including WIGOS compliance, complete station history, discovery metadata, and metadata quality validation

  • User Interface (Recommended): Interactive visualization, data review, and quality control validation capabilities

  • Climate Products (Recommended): Production and publication of standard climate products and indices

  • Processing Automation (Recommended): Scheduled and automated execution of routine data processing workflows

Note

The classification of conformance classes as Requirement, Recommendation and Permission aligns with the Required, Recommended and Optional classifications respectively from WMO-No. 1131. Required classes must be implemented for a system to be considered conformant. Recommended classes should be implemented where operationally feasible. Optional classes may be implemented based on specific organizational needs.

The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in IETF RFC 2119.

The keywords indicate the level of strictness required for compliance with a normative provision:

  • MUST/REQUIRED/SHALL: These keywords mean that the definition is an absolute requirement of the specification.

  • MUST NOT/SHALL NOT: These phrases mean that the definition is an absolute prohibition of the specification.

  • SHOULD/RECOMMENDED: These keywords mean that there may exist valid reasons in particular circumstances to ignore a particular item, but the full implications must be understood and carefully weighed before choosing a different course.

  • SHOULD NOT/NOT RECOMMENDED: These phrases mean that there may exist valid reasons in particular circumstances when the behavior is acceptable or even useful, but the full implications should be understood and the case carefully weighed before proceeding.

  • MAY/OPTIONAL: These keywords mean that an item is truly optional.

6. Functional Requirements

6.1. Requirements Class: Core

Requirements class

/req/core

Target type

Software system

Dependency

None

6.1.1. Overview

The Core requirements class defines the requirements for basic observation storage, retrieval, and metadata management capabilities. This provides the foundational data management capabilities for a Climate Data Management System and support for long-term data stewardship.

6.1.2. Justification and Regulatory Basis

WMO-No. 1131 identifies Core Data Management as a Required component across all CDMS implementations.

The Manual on High-quality Global Data Management Framework for Climate (WMO-No. 1238) requires Members to "ensure that Entities establish and document data management capabilities and practices for their climate data" (3.1.1). The framework’s principles mandate "preservation and management of all data and metadata that are used for, or may potentially be useful for, climate studies" (1.1.1(3)). Core capabilities support the FAIR data principles requiring data to be Findable, Accessible, Interoperable and Reusable (3.2).

6.1.3. Requirements

6.1.3.1. Observation Data Model

Requirement 1

Identifier

/req/core/data-model

Statement

The system SHALL implement a data model capable of storing in-situ point observations from across the Earth system domains and disciplines. The data model SHALL include:

  • Station and sensor metadata

  • Observed element/variable

  • Observation value/result and associated units

  • Quality flag status

  • Observation timestamp, including time zone support

  • Geographic location of the observation

  • Provenance metadata
Note

A draft data model <link to wccdm-obs>
Note

The definition of an observed element requires careful consideration of all dimensions that distinguish one measurement from another, including: measurement type (e.g. temperature, pressure), sensor height or depth, aggregation duration, statistical function (instantaneous, mean, maximum, minimum), and unit of measure. Systems should ensure that element definitions are sufficiently granular to avoid accidental overwriting of observations from different sensors or measurement configurations. Source data provenance (e.g. transmission format or network) should be recorded separately rather than embedded in the element definition.
Note

This specification focuses primarily on fixed-position stations. Observations from mobile platforms (ships, buoys, radiosondes, aircraft) may be ingested and archived, but specific requirements for real-time tracking or operational processing of mobile platforms are outside the scope of this specification.

6.1.3.8. Provenance Metadata Storage

Requirement 7

Identifier

/req/core/provenance-storage

Statement

The system SHALL store provenance metadata for all observation data documenting the complete data lineage including:

  • Original data source identification (network, station, transmission method)

  • Acquisition timestamp (when data entered the system)

  • Ingestion processing details (format conversion, unit normalization, coordinate transformation)

  • Quality control processing history (quality control procedures applied, flags assigned, timestamps)

  • Manual interventions (edits, corrections, deletions with operator identification and justification)

  • Derived data lineage (source observations, algorithms, parameters used in derivation)

  • Homogenisation adjustments (break points detected, adjustment factors applied, reference series used)

  • Data rescue provenance (source document reference, image identifier, keying method, verification status)
6.1.3.9. Provenance Metadata Retrieval

Requirement 8

/req/core/provenance-retrieval

Statement

The system SHALL support retrieval of provenance metadata including:

  • Identification of all transformations applied to data from acquisition to current state

  • Audit trail queries by operator, time period, or processing type

  • Traceability from derived products back to original source observations

  • Export of provenance information in machine-readable format (e.g., W3C PROV-O, ISO 19115 lineage)
6.1.3.10. Provenance Metadata Immutability

Requirement 9

Identifier

/req/core/provenance-immutability

Statement

The system SHALL ensure provenance records are immutable such that:

  • Provenance entries cannot be modified or deleted after creation

  • A complete audit trail exists for all data modifications

Requirement 10

  • Progress indicators for long-running operations

6.2. Requirements Class: Observations Quality Control

Requirements class

/req/quality-control

Target type

Software system

Dependency

/req/core

6.2.1. Overview

The Observations Quality Control requirements class defines requirements to ensure observation data meets quality standards for long-term environmental records through systematic automated checks that detect erroneous, suspicious, or inconsistent values while maintaining complete audit trails of all quality control decisions. Quality control is essential for establishing trustworthy observational records suitable for long-term trend analysis and monitoring.

6.2.2. Justification and Regulatory Basis

WMO-No. 1131 classifies Observations Quality Control as Required (Section 5.3), emphasizing that "all data have been checked for accuracy, and that a record is maintained of what checks were performed, the outcome of the checks and when the checks took place."

WMO-No. 1238 mandates that "Entities apply quality control to all climate datasets under their management and record the result of that quality control in the metadata" (3.3.2.1).

Note

The Guidelines on Surface Station Data Quality Control and Quality Assurance for Climate Applications (WMO-No. 1269) provide specific methodologies referenced in both documents.

6.2.3. Requirements

6.2.3.1. Quality Flag Management

Requirement 11

Identifier

/req/quality-control/flags

Statement

The system SHALL maintain quality flags and:

  • Support multiple quality control levels (automatic, manual, final)

  • Preserve quality control history and audit trail

  • Allow manual override with justification

  • Record which specific quality control check(s) triggered each flag and the reason for failure
Note

Implementers are encouraged to adopt a published flagging convention to facilitate data exchange and interoperability. WMO-No. 1269 provides guidance on quality flag definitions applicable to long-term observational data.
6.2.3.2. Quality Control Configuration Management

Requirement 12

Identifier

/req/quality-control/configuration

Statement

The system SHALL support configuration of quality control parameters, including:

  • Station-specific and element-specific thresholds

  • Seasonal variation of thresholds where climatologically appropriate

  • Enabling/disabling of individual checks per station or element

  • Version control of quality control rule configurations with effective dates
6.2.3.3. Quality Control Execution

Requirement 13

Identifier

/req/quality-control/execution

Statement

The system SHALL support triggering of quality control processing:

  • Automatically upon data ingestion

  • On demand by an authorized user

  • As a scheduled or batch process, including reprocessing of previously checked observations
6.2.3.4. Range Checks

Requirement 14

Identifier

/req/quality-control/range-checks

Statement

The system SHALL perform range validation checks, including:

  • Gross error limits (physically impossible values)

  • Climatological based range checks

  • Sensor-based range checks (measurements within observable range for sensor)

  • Configurable thresholds per element and station
6.2.3.5. Temporal Consistency Checks

Requirement 15

Identifier

/req/quality-control/temporal-checks

Statement

The system SHALL perform temporal consistency checks, including:

  • Flatline checks - detection of repeated identical values

  • Rate of change checks - validation of temporal gradients

  • Spike checks - detection of anomalous rapid changes followed by rapid reversals
6.2.3.6. Spatial Consistency Checks

Requirement 16

Identifier

/req/quality-control/spatial-checks

Statement

The system SHALL perform spatial consistency checks including:

  • Comparison with neighbouring stations

  • Spatial regression tests

  • Buddy checks using configurable distance and elevation thresholds
6.2.3.7. Internal Consistency Checks

Requirement 17

Identifier

/req/quality-control/internal-consistency

Statement

The system SHALL perform internal consistency checks to validate physical relationships between simultaneously observed elements as specified in WMO-No. 1269.
Note

Examples of internal consistency checks include validation of temperature relationships (Tmin ≤ Tavg ≤ Tmax), humidity constraints, pressure consistency, and precipitation-weather code agreement. WMO-No. 1269 provides comprehensive guidance on element-specific checks.
Note

Some internal consistency checks may also be applied within Data Collection and Processing Systems (DCPS) during real-time processing. The CDMS applies these checks in the context of quality assurance for long-term observational records, where access to full station metadata and the complete observation history allows for more thorough validation than is typically possible in real-time operational systems.

6.3. Requirements Class: Data Ingestion

Requirements class

/req/data-ingestion

Target type

Software system

Dependency

/req/core

6.3.1. Overview

The Data Ingestion requirements class defines requirements for the reliable ingestion of observational data from diverse sources and formats. This requirements class is restricted to observational (in-situ) data from stations and does not cover the ingestion of gridded or raster data products (e.g., satellite imagery, reanalysis fields, radar composites). Data ingestion serves as the entry point for observations into the observational record, requiring careful handling to preserve data integrity through validation and normalization while maintaining complete provenance of source data characteristics.

6.3.2. Justification and Regulatory Basis

WMO-No. 1131 identifies Data Ingest as a Required capability (Section 5.1.1), stating systems must handle "WMO messages, vector, raster array, ASCII and other formats" with "business rules, status logging, automated self-recovery and transformation." This specification restricts the scope of data ingestion to observational (in-situ) data formats; raster and gridded data ingestion is outside the scope of this specification.

WMO-No. 1238 requires Members ensure "Entities encode data, and associated metadata, intended for international exchange following the standards specified in the Manual on Codes (WMO-No. 306)" (3.2.2.1).

6.3.3. Requirements

6.3.3.1. Automated Data Collection

Requirement 18

Identifier

/req/data-ingestion/automated-collection

Statement

A

The system SHALL support the following data collection triggers:

  • Scheduled polling: Configurable polling of external data sources at defined intervals

  • Real-time subscription: Subscription to message brokers (e.g., MQTT) for event-driven data collection

  • File system monitoring: Automatic detection and ingestion of new files in monitored directories

6.7. Requirements Class: Metadata Management

Requirements class

/req/metadata-management

Target type

Software system

Dependency

/req/core, /req/security

6.7.1. Overview

Maintain comprehensive metadata supporting data interpretation, quality control, homogenisation, and fitness-for-purpose assessment. This encompasses station and sensor metadata (WIGOS), formal definitions of observable elements, dataset-level discovery and lineage metadata, dataset quality reporting, access constraints, and acquisition provenance. Comprehensive metadata is essential for establishing the context and trustworthiness of observational records.

6.7.2. Justification and Regulatory Basis

WMO-No. 1131 classifies Climate Metadata management as Required (Section 4.3 and 5.5), encompassing "observations metadata, discovery metadata, and data provenance metadata."

WMO-No. 1238 requires Entities to create and maintain three types of metadata: discovery metadata (per WMO-No. 1060, Volume II), observations metadata (per WMO No. 49, Volume I), and provenance metadata, defined as "information documenting the origin, processing history, and all modifications to a dataset throughout its lifecycle."

For findability and discovery (3.2.1.1), Entities SHALL assign globally unique and persistent identifiers to data and metadata (a), describe data with WIS-conformant metadata (b), and ensure data and metadata are registered in the WMO Information System Global Discovery Catalogue (c). For reusability (3.2.3), Entities SHALL include a usage licence with recommended data published through WIS (3.2.3.2), and any licence for core data SHALL be compatible with the WMO Unified Data Policy (3.2.3.3).

Section 3.3.1 establishes provenance requirements: Entities SHOULD maintain comprehensive records of all modifications and processing throughout the data lifecycle (3.3.1.1), including the origin of observations, processing history, and methodology references (3.3.1.2). Entities SHALL collect observational metadata and include it in provenance records (3.3.1.4), with reference to the WIGOS Metadata Standard (WMO-No. 1192) and the Manual on the WMO Integrated Global Observing System (WMO-No. 1160). Entities SHALL include a full version history for critical datasets used to monitor climate change (3.3.1.6). Section 2.1.2.3 further requires that Entities responsible for making observations follow the provisions for collection, management, and publication of observational metadata as specified in WMO-No. 1160.

Section 3.3.2 establishes quality assurance metadata requirements: Entities SHALL apply quality control to all datasets and record results in metadata (3.3.2.1), SHALL assign quality flags or indicators to all records (3.3.2.2), SHOULD update metadata with assessed quality at each stage of the quality control process (3.3.2.3), and SHALL record any homogenisation or reprocessing in provenance metadata with links to processing documentation (3.3.2.5).

The ISO 19115 family of standards (ISO 19115-1, ISO 19115-2, ISO 19157, ISO 19110) provides widely recognised concepts for geographic dataset metadata, including lineage, data quality, constraints, maintenance, and feature cataloguing. These concepts inform the requirements in this class; however, as ISO standards are not freely available, they are referenced informatively rather than as normative dependencies. Freely available standards are used as normative references throughout, including the WMO WIGOS Metadata Standard (WMO-No. 1192), WCMP2, W3C PROV-O (provenance), and the CF Conventions for Climate and Forecast Metadata.

6.7.3.4. Discovery Metadata

Requirement 39

Identifier

/req/metadata-management/discovery

A

The system SHALL maintain dataset-level discovery metadata records compliant with the WMO Core Metadata Profile version 2 (WCMP2). Each record SHALL include all mandatory WCMP2 properties:

  • A unique record identifier following the URN pattern urn:wmo:md:{centre-id}:{local-identifier}

  • Resource title and free-text description

  • Spatial extent (bounding box in WGS84) and temporal extent

  • Earth system discipline classification using the WIS2 Topic Hierarchy vocabulary (properties.themes)

  • Responsible organisation name and contact role (properties.contacts)

  • WMO data policy classification (core or recommended) in properties.wmo:dataPolicy

  • At least one link to a data access endpoint

B

The system SHALL maintain dataset lineage metadata following the W3C PROV Data Model, including:

  • A lineage statement summarising the overall processing history of the dataset

  • Source data references: originating station networks, instruments, and observation periods

  • A record for each significant processing stage applied to the dataset, covering at minimum:

    • Ingestion and format normalisation

    • Quality control procedures and the version or configuration applied

    • Homogenisation adjustments, where applicable

    • Statistical aggregation or derived product computation, where applicable

  • For each processing stage: date, description, responsible party, and software or method reference

C

The system SHALL support submission and update of WCMP2 records to the WIS2 Global Discovery Catalogue (GDC), either directly or via a configured WIS2 Node.
Note

WCMP2 is built on OGC API - Records (GeoJSON) and intentionally does not incorporate the ISO 19115/19139 XML structures used in earlier WMO metadata standards. WCMP2 serves the discovery function: making datasets findable in the WIS2 Global Discovery Catalogue. Dataset lineage metadata, structured following the W3C PROV Data Model (https://www.w3.org/TR/prov-dm/), serves a complementary function: documenting the detailed processing provenance of datasets for scientific interpretation and fitness-for-purpose assessment. The PROV Data Model defines the core concepts of Entity (the dataset or observation series), Activity (a processing step), and Agent (the responsible party or software).

Discovery metadata records are dataset-level descriptions and are distinct from the station and sensor metadata described in /req/metadata-management/wigos.

Two levels of provenance/lineage are maintained in a CDMS: (1) observation-level provenance (/req/core/provenance-storage, /req/core/provenance-immutability) records the complete immutable history of each individual observation — ingestion, QC flags, corrections, and data rescue events; (2) dataset-level lineage (this requirement, Statement B) summarises the processing history of a collection of observations as a whole, supporting discovery and fitness-for-purpose assessment. Dataset-level lineage is a descriptive summary; it is not a substitute for the immutable observation-level provenance record and does not carry the same immutability constraint.

Export formats are addressed in /req/metadata-management/export.

For datasets classified as recommended under the WMO data policy, at least one access link with associated licence information is required by WCMP2.

The concepts of lineage statement, process step, and source data used in Statement B above are informed by the ISO 19115 family (ISO 19115-1 LI_Lineage, ISO 19115-2 LE_ProcessStep); however, those standards are not freely available and are not normative dependencies of this specification.

7.2.1. Overview

Ensure the system operates dependably with minimal unplanned downtime, maintains data integrity under all conditions, and recovers gracefully from failures. Reliability is essential for climate data, which serves as an irreplaceable scientific resource requiring protection for future generations.

7.2.2. Justification and Regulatory Basis

WMO-No. 1131 states that "the most important stakeholders for climate data are future generations, and the reason to implement and manage CDMSs now is to protect the integrity of climate data for their future use" (Section 2.3). The specification’s core theme is to "ensure that the data are securely and transparently managed" (Section 2.3).

WMO-No. 1238 mandates that "Members shall ensure that Entities develop and maintain a business continuity plan to mitigate risks associated with disruption of operations" (3.4.1.1) and requires "routine backup, and procedures for timely restoration of their databases" (3.4.1.2). The requirement to "retain their original and irreproducible climate records, and associated metadata, permanently" (3.4.1.4) establishes reliability as a foundational requirement.

7.2.3.1. System Availability

Identifier

/req/reliability/availability

Statement

  • Define and publish availability targets for data retrieval and ingestion services

  • Monitor system availability and report against defined targets

  • Communicate scheduled maintenance windows to users in advance

  • Minimise scheduled maintenance downtime through rolling update or equivalent procedures

  • Notify administrators promptly of unplanned outages with sufficient detail for diagnosis
Note

Retrieval availability is typically held to a higher target than ingestion availability, since brief ingestion interruptions are often recoverable (source data can be buffered and re-submitted) while retrieval unavailability directly affects users. Implementers should set targets appropriate to their national operational context and infrastructure. Note that availability targets govern day-to-day operations; disaster recovery objectives (RTO/RPO) are addressed separately in /req/reliability/backup-recovery.
7.2.3.2. Data Integrity

Requirement 59

Identifier

/req/reliability/data-integrity

Statement

The system SHALL ensure data integrity through:

  • ACID-compliant transactions for all data create, update, and delete operations

  • Referential integrity constraints preventing orphaned observations (observations without valid station references)

  • Checksums or cryptographic hash verification for file-based data transfers

  • Write-ahead logging or equivalent mechanisms to prevent data loss during system failures

  • Periodic integrity checks to detect and report data corruption
Note

Write-ahead logging (WAL) is a standard database durability mechanism in which changes are written to a transaction log before being applied to the main data store. In the event of a system crash, the log enables recovery to a consistent state. The specific mechanism is implementation-dependent; the requirement is that the system can recover to a consistent state following an unplanned failure.

Periodic integrity checks refer to automated verification that stored data matches expected checksums or hash values, detecting silent corruption caused by storage media failures or software defects. Results should be logged and administrators alerted on any detected inconsistency.

Referential integrity should be enforced as a system invariant. However, implementations receiving data from operational feeds (e.g., GTS, WIS 2.0) may encounter observations for stations not yet registered in the metadata repository. In such cases, implementations may auto-create a minimal station record pending metadata completion, rather than rejecting the observation. This exception should be logged and flagged for follow-up. See also /req/data-ingestion/validation.
7.2.3.3. Backup and Recovery

Requirement 60

Identifier

/req/reliability/backup-recovery

Statement

The system SHALL implement backup and recovery capabilities:

  • Automated regular backups of all observation data, metadata, and system configuration

  • Defined and documented Recovery Time Objective (RTO) and Recovery Point Objective (RPO) appropriate to the operational context

  • Point-in-time recovery capability to restore the system to a known-good prior state

  • Documented and regularly tested recovery procedures

  • Verification of backup integrity through periodic restore testing
Note

RTO defines the maximum acceptable time to restore operations following a failure. RPO defines the maximum acceptable data loss measured in time. Implementers should define targets appropriate to their national context — for a national CDMS, a pragmatic RPO will often be longer than for a real-time operational system, since historical observation data is typically re-ingestible from source systems. The critical concern is protecting processed outputs (QC flags, homogenisation adjustments) that may not be reproducible without significant effort.

Backup storage volumes grow with retention period. Implementers should balance retention requirements against available storage, using incremental backup strategies and compression to manage costs. WMO-No. 1238 requires permanent retention of original and irreproducible climate records (3.4.1.4); backup policies should reflect this for long-term archival.

7.6.1. Overview

Ensure the system can exchange data and integrate with external systems using industry-standard protocols, formats, and conventions. Interoperability enables climate data to flow seamlessly between national systems, regional centers, international archives, and the WIS 2.0 infrastructure.

7.6.2. Justification and Regulatory Basis

WMO-No. 1131 emphasizes that "open spatial standards are a mechanism that is being increasingly adopted by many organizations and industry sectors around the world" (Section 8.1) and that these standards are "expected to become an increasingly important mechanism for distributing data in future years." The specification notes that standardized delivery "will ensure that data are represented in a form that facilitates data interoperability and are accessible to a wide range of end users from disparate industries" (Section 3.1.4.1).

WMO-No. 1238 requires "Members shall ensure that Entities encode data, and associated metadata, intended for international exchange following the standards specified in the Manual on Codes (WMO-No. 306), the Manual on the WMO Information System (WMO-No. 1060)" (3.2.2.1). The framework principle mandating registration in "the WIS Global Discovery Catalogue" (1.1.1(4)) requires standards-based interoperability.

7.7. Requirements Class: Data Governance

Requirements class

/req/data-governance

Target type

Software system

Dependency

/req/core

7.7.1. Overview

Ensure proper stewardship of climate data through lifecycle management, quality documentation, access policies, and compliance with organizational and regulatory requirements. Data governance provides the policy framework and accountability structures necessary for long-term climate data stewardship.

7.7.2. Justification and Regulatory Basis

WMO-No. 1131 dedicates Chapter 3 entirely to "CDMS Governance," stating that "one of the steps often overlooked when implementing a CDMS is to ensure that the NMHS has established a solid governance framework to protect the integrity of the climate record" involving "Policies" and "Governance processes." The specification defines data governance as Required (Section 3.2.1), including "controlled access to data and systems," "approval process for new data types," "approval process to change data," and "IT change approvals.".

WMO-No. 1238 requires "Members should ensure that Entities implement and maintain a governance mechanism to oversee the various governance elements, coordinated by a governance body" (3.5.1.1) and mandates policies covering commitments, sustainability, intellectual property, and data delivery (Section 3.1 of WMO-No. 1131). The framework emphasizes that "data stewardship assigns to an entity rights and responsibilities for acquiring and managing climate data and information" (Definitions), requiring formalized governance structures.

7.7.3. Requirements

7.7.3.1. Data Lifecycle Management

Requirement 80

Identifier

/req/data-governance/lifecycle

Statement

The system SHALL support data lifecycle management with:

  • Configurable data retention policies per dataset type or network

  • Automated archival of historical data to long-term storage

  • Data deletion workflows with approval requirements for permanent deletion

  • Migration paths for data format updates or schema changes
7.7.3.2. Data Quality Metadata

Requirement 81

Identifier

/req/data-governance/quality-metadata

Statement

The system SHALL maintain data quality metadata including:

  • Completeness metrics (percentage of expected observations received)

  • Timeliness metrics (delay between observation time and system receipt)

  • Quality control statistics (pass/fail rates per check type)

  • Manual intervention frequency and types

  • Homogenization status and adjustment statistics

  • Overall fitness-for-purpose assessments

B

Quality metadata SHALL be:

  • Queryable alongside observation data

  • Exportable in machine-readable formats

  • Updated automatically as data quality changes
Note

Fitness-for-purpose refers to an assessment of whether a dataset meets the quality requirements of a particular application. For example, a dataset with frequent short gaps may be suitable for climatological trend analysis but unsuitable for computing climate normals. Fitness-for-purpose assessments summarise quality metrics (completeness, temporal continuity, QC flag distribution) in terms of suitability for defined use cases, supporting informed data selection by users. See also /req/metadata-management/data-quality.

7.7.3.4. Data Stewardship Roles

Requirement 83

Identifier

/req/data-governance/stewardship

Statement

The system SHALL define and support data stewardship roles including:

  • Data custodian responsibilities for dataset maintenance

  • Data quality manager authority for quality control decisions

  • Data access coordinator for managing access requests and policies

  • System administrator separation from data stewardship roles

  • Clear escalation paths for data governance issues
Note

Data stewardship roles define the organisational responsibilities for data management and are distinct from the technical access control roles defined in /req/security/authorization. In practice, stewardship roles map to RBAC roles: for example, the data quality manager role is associated with permissions to override quality control flags and approve homogenisation adjustments. The RBAC configuration in the security class provides the technical enforcement of the responsibilities defined here.
