=== Requirements Class: Metadata Management

[cols="1,4"]
|===
|Requirements class|/req/metadata-management
|Target type|Software system
|Dependency|/req/core, /req/security
|===

==== Overview

Maintain comprehensive metadata supporting data interpretation, quality control, homogenization, and
fitness-for-purpose assessment. This encompasses station and sensor metadata (WIGOS), formal definitions
of observable elements, dataset-level discovery and lineage metadata, dataset quality reporting, access
constraints, and acquisition provenance. Comprehensive metadata is essential for establishing the context
and trustworthiness of observational records.

[NOTE]
====
Many of the concepts used in this requirements class — including dataset lineage, data quality,
access constraints, and feature cataloguing — are in common use in geographic information management.
Readers familiar with the ISO 19115 family of standards will recognize these concepts. ISO standards
are not freely available and are not requirements of this specification.
====

==== Requirements

[[req_metadata_wigos_compliance]]
===== WIGOS Metadata Compliance

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/metadata-management/wigos
|Statement a|
|A a|The system SHALL support metadata fields as specified in WIGOS Metadata Standard (WMO-No. 1192), including:

* Station identification (WIGOS Station Identifier, national identifiers, historical identifiers)
* Station name, description, and alternative names
* Geographic location (latitude, longitude, elevation, datum reference)
* Administrative information (operating organization, country, region)
* Station classification (WMO station type, network affiliations, programme participation)
* Sensor and instrument inventory with make, model, serial number, and calibration status
* Measurement configuration (observed elements, measurement height/depth, sampling interval)
* Siting environment description and exposure classification
* Operating schedule and reporting practices
* Responsible party and contact information

|B a|The system SHALL support extensibility mechanisms to store metadata fields not covered by the
WIGOS Metadata Standard, including national identifiers, legacy identifiers from predecessor
networks, and other organization-specific metadata attributes.
|===

[[req_metadata_history]]
===== Station History Documentation

[cols="1,4"]
|===
2+^|Requirement {counter:req-id}
|Identifier|/req/metadata-management/history
|Statement a|The system SHALL:
|A a|maintain a complete station history, including:

* Station relocations with old and new coordinates
* Instrument changes with make, model, and installation dates
* Changes to observation practices or measurement heights
* Changes to surrounding environment affecting exposure
* Periods of station closure or data gaps
* Network affiliation changes

|B a|for each entry in the station history include:

* Effective date and time of the change
* Description of the change
* Reason for the change where known
* Responsible party documenting the change
* Supporting documentation references
|===

[NOTE]
====
The station history (Statement A) combined with the station and sensor metadata maintained under
/req/metadata-management/wigos provides full dataset-to-instrument traceability: for any observation
timestamp, the active instrument configuration can be determined from the instrument change records
and their effective dates. This information supports reproducibility and uncertainty assessment without
requiring a separate acquisition metadata record.
====

[[req_metadata_feature_catalogue]]
===== Element and Feature Catalogue

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/metadata-management/feature-catalogue
|Statement a|The system SHALL maintain a catalogue of observable elements, recording for each element:

* A unique element identifier and human-readable name
* Measurement class (e.g., temperature, precipitation, pressure, wind speed)
* Accumulation or averaging duration
* Unit of measure
* Statistical function (instantaneous, mean, maximum, minimum, accumulated total, etc.)
* Applicable value range and resolution
* Associated controlled vocabulary reference: a WMO BUFR parameter code (from the freely available
  WMO code tables) or a CF standard name (from the freely available CF Conventions) where applicable
|===

[[req_metadata_discovery]]
===== Discovery Metadata

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/metadata-management/discovery
|A a|The system SHALL maintain dataset-level discovery metadata records compliant with the WMO Core
Metadata Profile version 2 (WCMP2). Each record SHALL include all mandatory WCMP2 properties:

* A unique record identifier following the URN pattern `urn:wmo:md:{centre-id}:{local-identifier}` (e.g., `urn:wmo:md:br-inmet:daycli`)
* Resource or dataset title and free-text description
* Spatial extent (bounding box in WGS84) and temporal extent
* Earth system discipline classification using the WIS2 Topic Hierarchy vocabulary
  (`properties.themes`)
* Responsible organization name and contact role (`properties.contacts`)
* WMO data policy classification (`core` or `recommended`) in `properties.wmo:dataPolicy`
* At least one link to a data access endpoint

|B a|The system SHALL maintain dataset lineage metadata following the W3C PROV Data Model,
including:

* A lineage statement summarizing the overall processing history of the dataset
* Source data references: originating station networks, instruments, and observation periods
* A record for each significant processing stage applied to the dataset, covering at minimum:
** Ingestion and format normalization
** Quality control procedures and the version or configuration applied (more than one record may be required where the dataset undergoes multiple stages of quality control, e.g. automated checks followed by manual or expert-system review)
** Homogenization and other adjustments, where applicable
** Disaggregation or infilling, where applicable
** Statistical aggregation or derived product computation, where applicable
* For each processing stage: date, description, responsible party, and software or method reference

|===

[NOTE]
====
WCMP2 is built on OGC API - Records (GeoJSON) and intentionally does not incorporate the ISO 19115/19139
XML structures used in earlier WMO metadata standards. WCMP2 serves the discovery function: making
datasets findable in the WIS2 Global Discovery Catalogue. Dataset lineage metadata, structured
following the W3C PROV Data Model (https://www.w3.org/TR/prov-dm/), serves a complementary function:
documenting the detailed processing provenance of datasets for scientific interpretation and
fitness-for-purpose assessment. The PROV Data Model defines the core concepts of Entity (the dataset
or observation series), Activity (a processing step), and Agent (the responsible party or software).

Discovery metadata records (WCMP2) are dataset-level descriptions that make data findable in WIS2 —
they describe _what_ data exists and _where_ to access it. They are entirely distinct from station
and sensor metadata (/req/metadata-management/wigos), which describe the observing platform and
instruments that produced the data. Confusing these two metadata types is a common source of
implementation errors.

Two levels of provenance/lineage are maintained in a CDMS: (1) observation-level provenance
(/req/core/provenance-storage, /req/core/provenance-immutability) records the complete immutable
history of each individual observation — measurement method and instrumentation, ingestion, QC flags, corrections, and data rescue events;
(2) dataset-level lineage (this requirement, Statement B) summarizes the processing history of a
collection of observations as a whole, supporting discovery and fitness-for-purpose assessment.
Dataset-level lineage is a descriptive summary; it is not a substitute for the immutable
observation-level provenance record and does not carry the same immutability constraint.

Export formats are addressed in /req/metadata-management/export.

For datasets classified as `recommended` under the WMO data policy, at least one access link with
associated licence information is required by WCMP2.

[[req_metadata_data_quality]]
===== Dataset Quality Reporting

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/metadata-management/data-quality
|A a|The system SHALL maintain dataset-level quality metadata including:

* Completeness measures: proportion of expected observations present over the dataset period,
  covering both omission (records absent) and commission (duplicate or spurious records)
* Temporal quality: consistency of timestamps and identification of temporal gaps or overlaps
* Thematic accuracy: conformance of values to expected ranges and inter-element consistency,
  drawing on quality control results from /req/quality-control
* A conformance result indicating whether the dataset meets defined quality thresholds

|B a|The system SHALL generate dataset quality reports that:

* Summarize completeness by station, element, and time period
* Identify temporal gaps and periods of reduced data availability
* Report the distribution of quality control flag statuses across the dataset
|===

[NOTE]
====
Dataset-level quality reporting is distinct from per-observation quality flags managed by
/req/quality-control. Quality flags on individual observations provide the source data for computing
dataset-level quality metrics. These aggregated metrics support fitness-for-purpose assessment, inform users about the
suitability of a dataset for specific applications, and should help inform observation providers
about any systemic errors or patterns of errors within the dataset.

Dataset quality metrics may be derived automatically from per-observation quality control results,
or entered manually by data managers to document known issues or provide additional context.
Dataset quality reporting (Statement B) covers the generation and export of reports drawing on
both sources.

[[req_metadata_constraints]]
===== Data Access Constraints

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/metadata-management/constraints
|Statement a|The system SHALL record data access and use constraints for each dataset, including:

* Legal access constraints: any restrictions on who may access the data
* Use constraints: conditions on redistribution or derivative use
* Security classification where applicable (e.g., national security requirements)
* Embargo periods: dates before which data may not be published externally
|===

[cols="1,4"]
|===
2+^|Recommendation {counter:rec-id}
|Identifier|/rec/metadata-management/constraints
|Statement a|The system SHOULD record the WMO data policy classification (core or recommended) for
each dataset, consistent with the `wmo:dataPolicy` property specified in WCMP2 and WMO data policy
requirements defined in WMO-No. 1238.
|===

[NOTE]
====
The constraints in this requirement apply at dataset or network level, not to individual observations.

Constraint metadata must be enforced through the access control mechanisms defined in
/req/security/authorization. Embargo periods and redistribution restrictions recorded here
should be applied consistently across all data access interfaces, including the API
(/req/data-api) and any export functions.
====

[[req_metadata_export]]
===== Metadata Standards Export

[cols="1,4"]
|===
2+^|Requirement {counter:req-id}
|Identifier|/req/metadata-management/export
|Statement a|The system SHALL support export of metadata in standard formats including:

* WIGOS Metadata Representation (WMDR) for station and sensor metadata
* WMO Core Metadata Profile v2 (WCMP2, GeoJSON) for dataset discovery metadata
* W3C PROV-O or equivalent structured format for dataset lineage and provenance records
* Dataset quality reports in a documented, structured format (e.g., JSON or XML with
  defined schema) covering completeness, temporal quality, and quality flag distribution
* Element catalogue in a documented structured format referencing WMO BUFR parameter codes
  or CF standard names
* Station metadata exchange formats used by regional or global programmes
|===

[NOTE]
====
A simplified second version of the WIGOS Metadata Representation (WMDR) is under development.
Further alignment with the updated version is anticipated when published.
====

[cols="1,4"]
|===
2+^|Requirement {counter:req-id}
|Identifier|/req/metadata-management/validation
|Statement a|The system SHALL
|A a|validate metadata for:

* Required field completeness per WIGOS standards
* Coordinate system validity and range checks
* Temporal consistency (effective dates do not overlap)
* Referential integrity (stations referenced by observations exist and are correct)
* Controlled vocabulary compliance for classification fields

|B a|generate validation reports identifying:

* Missing required metadata
* Metadata quality issues or inconsistencies
* Recommendations for metadata improvement
|===
