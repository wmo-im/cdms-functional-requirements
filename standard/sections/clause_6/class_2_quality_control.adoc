=== Requirements Class: Observations Quality Control

[cols="1,4"]
|===
|Requirements class|/req/quality-control
|Target type|Software system
|Dependency|/req/core
|===

==== Overview

The Observations Quality Control requirements class defines requirements to ensure observation data meets quality
standards for long-term environmental records through systematic quality checks that detect erroneous, suspicious,
or inconsistent values while maintaining complete audit trails of all quality control decisions. Quality control is
essential for establishing trustworthy observational records suitable for long-term trend analysis and monitoring.
The results of quality control processing may also help identify systemic error patterns in the observations
that imply the need for remedial training or other actions (see /req/data-governance/stewardship),
and/or may inform design changes to the quality control algorithms if too few errors are detected
or too many values are flagged as suspect.

==== Justification and Regulatory Basis

WMO-No. 1131 classifies Observations Quality Control as Required (Section 5.3), emphasizing that "all data have been
checked for accuracy, and that a record is maintained of what checks were performed,
the outcome of the checks and when the checks took place."

WMO-No. 1238 requires Entities to apply quality control to all climate datasets under their management and record
the results in the associated metadata (Section 3.3.2).

[NOTE]
====
The  __Guidelines on Surface Station Data Quality Control and Quality Assurance for Climate Applications__
(WMO-No. 1269) provide specific methodologies referenced in both of the above documents.
====

==== Requirements

[[req_qc_observation_retention]]
===== Observation Retention

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/observation-retention
|Statement a|The system SHALL retain all observation records regardless of quality flag status.
Observations that fail quality control checks SHALL remain in the system with their assigned
quality flags and the complete record of checks applied.
|===

[[req_qc_flag_management]]
===== Quality Flag Management

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/flags
|Statement a|
The system SHALL maintain quality flags and:

* Support multiple quality control levels (automatic, manual, final)
* Preserve quality control history and audit trail
* Allow manual override with justification
* Record which specific quality control check(s) triggered each flag and the reason for failure

|===

[NOTE]
====
Implementers are encouraged to adopt a published flagging convention to facilitate data exchange and
interoperability. The _Guide to the Global Observing System_ (WMO-No. 488) defines basic quality control
flag categories, and WMO-No. 1269 provides further guidance on quality flag definitions applicable to
long-term observational data.
====

[[req_qc_configuration]]
===== Quality Control Configuration Management

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/configuration
|Statement a|
The system SHALL support configuration of quality control parameters, including:

* Station-specific and element-specific thresholds
* Seasonal variation of thresholds where climatologically appropriate
* Enabling/disabling of individual checks per station or element
* Version control of quality control rule configurations with effective dates

|===

[NOTE]
====
Thresholds may be either fixed values (e.g. absolute physical limits) or dynamically derived (e.g. based on
climatological statistics such as percentiles or standard deviations). Implementations should ensure that simple fixed-threshold
configurations remain straightforward to set up, with advanced options such as seasonally varying or
statistically derived thresholds available but not required for basic operation.
====

[[req_qc_execution]]
===== Quality Control Execution

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/execution
|Statement a|
The system SHALL support quality control processing initiated in multiple ways:

* Automatically upon data ingestion
* On demand by an authorized user, including reprocessing of previously checked observations
* As a scheduled or batch process, including reprocessing of previously checked observations
|===

[NOTE]
====
A series of standard checks is defined in the following sections. Not all checks are applicable to all
variables or observation frequencies; for instance, rate of change and spike checks are not relevant to
the quality control of daily accumulated rainfall.
====

[[req_qc_range_checks]]
===== Range Checks

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/range-checks
|Statement a|The system SHALL perform range validation checks, including:

* Gross error limits (physically impossible values, e.g. wind direction greater than 360 degrees or temperature greater than 80°C)
* Climatologically-based range checks
* Sensor-based range checks (measurements within observable range for sensor)
* Configurable thresholds per element and station
|===

[NOTE]
====
Element-specific gross error limits are defined in the _Guide to the Global Observing System_ (WMO-No. 488)
and the _Guide to Meteorological Instruments and Methods of Observation_ (WMO-No. 8) and should be used
as the basis for gross error limit checks where available.
====

[[req_qc_temporal_consistency]]
===== Temporal Consistency Checks

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/temporal-checks
|Statement a|The system SHALL perform temporal consistency checks, including:

* Flatline checks - detection of repeated identical values
* Rate of change checks - validation of temporal gradients
* Spike checks - detection of anomalous rapid changes followed by rapid reversals
|===

[[req_qc_spatial_consistency]]
===== Spatial Consistency Checks

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/spatial-checks
|Statement a|The system SHALL perform spatial consistency checks including:

* Comparison with neighbouring observations using configurable selection criteria, including
  maximum distance, elevation difference, station type, covariability between stations,
  and predefined station lists
* Spatial regression tests
|===

[NOTE]
====
Where no neighbouring stations exist within a practicable distance — as is the case for many island and
remote stations — spatial consistency checks cannot be applied. The maximum distance threshold used for
neighbour selection should be configurable to reflect regional station density.
====

[[req_qc_internal_consistency]]
===== Internal Consistency Checks

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/quality-control/internal-consistency
|Statement a|The system SHALL perform internal consistency checks to validate physical relationships between simultaneously observed elements as specified in WMO-No. 1269.
|===

[NOTE]
====
Examples of internal consistency checks include validation of temperature relationships
(T~min~ {leq} T~avg~ {leq} T~max~), humidity constraints, pressure consistency,
and precipitation-weather code agreement. WMO-No. 1269 provides comprehensive
guidance on element-specific checks.
====

[NOTE]
====
Some internal consistency checks may also be applied within Data Collection and Processing Systems
(DCPS) during real-time processing. The CDMS applies these checks in the context of quality assurance
for long-term observational records, where access to the observation history allows for more thorough
validation than is typically possible in real-time operational systems.
====
