=== Requirements Class: Data Rescue

[cols="1,4"]
|===
|Requirements class|/req/data-rescue
|Target type|Software system
|Dependency|/req/core, /req/data-ingestion
|===

==== Overview

Facilitate the recovery of historical observations from non-digital sources, ensuring rescued data is
integrated into the observational records with complete provenance and appropriate quality control. Data rescue
extends the temporal coverage of observational records, essential for understanding long-term variability and change.

==== Requirements

[[rec_data_rescue_imaging]]
===== Document Imaging Management

[cols="1,4"]
|===
2+^|**Recommendation {counter:rec-id}**
|Identifier|/rec/data-rescue/imaging
|Statement a|
The system SHOULD support document imaging workflows including:

* Storage of scanned images with persistent identifiers
* Association of images with source document metadata (archive, collection, box number, date range)
* Support for multiple image formats (JPEG, TIFF, PDF)
* Image quality metadata (resolution, color depth, scan date)
* Linkage between images and digitized observations

|===

[NOTE]
====
Automated Optical Character Recognition (OCR) of historical documents is outside the scope of this
specification. The data entry workflows described here are based on manual keying by operators, with
source images displayed alongside data entry forms to support transcription. OCR tools may be used as
a supplementary aid at the implementer's discretion.
====

[[rec_data_rescue_interface]]
===== Data Entry Interface

[cols="1,4"]
|===
2+^|**Recommendation {counter:rec-id}**
|Identifier|/rec/data-rescue/data-entry
|A a|
The system SHOULD provide a data entry interface for manual keying that:

* Displays source images alongside data entry forms
* Supports configurable data entry templates based on historical form layouts
* Provides real-time validation of entered values
* Alerts the operator to gross errors that may result from keying mistakes, with the option to re-key the affected values
* Tracks data entry operator and timestamp
* Supports both single-key and double-key entry workflows for quality assurance
* Supports drag-and-drop interaction for assigning images to entry forms, particularly where the variety of historical forms is large

|===

[[req_data_rescue_qc]]
===== Data Rescue Quality Control

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/data-rescue/quality-control
|Statement a|The system SHALL apply quality control to rescued data including:

* Where double-key entry is used, comparison between entries to identify transcription errors
* Range and consistency checks appropriate to historical observation practices
* Verification against original images when discrepancies are detected
* Flagging of uncertain or illegible values
* Documentation of data rescue quality assurance decisions

|===

[NOTE]
====
Digitized observations are processed through the same quality control pipeline as contemporary
observations (/req/quality-control). The sequencing of digitization, provisional storage, and quality
assurance is an implementation decision; rescued data does not enter the operational observational
record without passing through the full QC pipeline.
====

[[req_data_rescue_provenance]]
===== Data Rescue Provenance

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/data-rescue/provenance
|Statement a|
The system SHALL record data rescue provenance including:

* Source document identification and archive location
* Image identifier and scan metadata
* Data entry operator(s) and keying date(s)
* Keying method used (e.g., single-key, double-key); automated OCR is outside the scope of this specification
* Quality assurance verification status
* Known limitations or uncertainties in rescued data
|===
