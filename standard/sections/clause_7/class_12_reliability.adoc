=== Requirements Class: Reliability and Availability

[cols="1,4"]
|===
|Requirements class|/req/reliability
|Target type|Software system
|Dependency|/req/core
|===

==== Overview

Ensure the system operates dependably with minimal unplanned downtime, maintains data integrity under all conditions,
and recovers gracefully from failures. Reliability is essential for climate data, which serves as an irreplaceable
scientific resource requiring protection for future generations.

==== Justification and Regulatory Basis

WMO-No. 1131 states that "the most important stakeholders for climate data are future generations, and the reason to
implement and manage CDMSs now is to protect the integrity of climate data for their future use" (Section 2.3).
The specification's core theme is to "ensure that the data are securely and transparently managed" (Section 2.3).

WMO-No. 1238 mandates that "Members shall ensure that Entities develop and maintain a business continuity plan to
mitigate risks associated with disruption of operations" (3.4.1.1) and requires "routine backup, and procedures for
timely restoration of their databases" (3.4.1.2). The requirement to "retain their original and irreproducible climate
records, and associated metadata, permanently" (3.4.1.4) establishes reliability as a foundational requirement.

==== Requirements

===== System Availability

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/reliability/availability
|Statement a|The system SHALL:

* Define and publish availability targets for data retrieval and ingestion services
* Monitor system availability and report against defined targets
* Communicate scheduled maintenance windows to users in advance
* Minimise scheduled maintenance downtime through rolling update or equivalent procedures
* Notify administrators promptly of unplanned outages with sufficient detail for diagnosis
|===

[NOTE]
====
Retrieval availability is typically held to a higher target than ingestion availability,
since brief ingestion interruptions are often recoverable (source data can be buffered
and re-submitted) while retrieval unavailability directly affects users. Implementers
should set targets appropriate to their national operational context and infrastructure.
Note that availability targets govern day-to-day operations; disaster recovery objectives
(RTO/RPO) are addressed separately in /req/reliability/backup-recovery.
====

===== Data Integrity

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/reliability/data-integrity
|Statement a|The system SHALL ensure data integrity through:

* ACID-compliant transactions for all data create, update, and delete operations
* Referential integrity constraints preventing orphaned observations (observations without valid station references)
* Checksums or cryptographic hash verification for file-based data transfers
* Write-ahead logging or equivalent mechanisms to prevent data loss during system failures
* Periodic integrity checks to detect and report data corruption
|===

[NOTE]
====
Write-ahead logging (WAL) is a standard database durability mechanism in which changes
are written to a transaction log before being applied to the main data store. In the
event of a system crash, the log enables recovery to a consistent state. The specific
mechanism is implementation-dependent; the requirement is that the system can recover
to a consistent state following an unplanned failure.

Periodic integrity checks refer to automated verification that stored data matches
expected checksums or hash values, detecting silent corruption caused by storage
media failures or software defects. Results should be logged and administrators
alerted on any detected inconsistency.

Referential integrity should be enforced as a system invariant. However, implementations
receiving data from operational feeds (e.g., GTS, WIS 2.0) may encounter observations
for stations not yet registered in the metadata repository. In such cases, implementations
may auto-create a minimal station record pending metadata completion, rather than
rejecting the observation. This exception should be logged and flagged for follow-up.
See also /req/data-ingestion/validation.
====

===== Backup and Recovery

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/reliability/backup-recovery
|Statement a|The system SHALL implement backup and recovery capabilities:

* Automated regular backups of all observation data, metadata, and system configuration
* Defined and documented Recovery Time Objective (RTO) and Recovery Point Objective (RPO)
  appropriate to the operational context
* Point-in-time recovery capability to restore the system to a known-good prior state
* Documented and regularly tested recovery procedures
* Verification of backup integrity through periodic restore testing
|===

[NOTE]
====
RTO defines the maximum acceptable time to restore operations following a failure.
RPO defines the maximum acceptable data loss measured in time. Implementers should
define targets appropriate to their national context — for a national CDMS, a pragmatic
RPO will often be longer than for a real-time operational system, since historical
observation data is typically re-ingestible from source systems. The critical concern
is protecting processed outputs (QC flags, homogenisation adjustments) that may not
be reproducible without significant effort.

Backup storage volumes grow with retention period. Implementers should balance
retention requirements against available storage, using incremental backup strategies
and compression to manage costs. WMO-No. 1238 requires permanent retention of original
and irreproducible climate records (3.4.1.4); backup policies should reflect this for
long-term archival.
====


===== Fault Tolerance

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/reliability/fault-tolerance
|Statement a|The system SHALL handle failures gracefully by:

* Continuing to serve read requests when write operations are unavailable
* Queuing ingestion requests during temporary processing unavailability
* Providing meaningful error messages rather than generic failures
* Logging all error conditions with sufficient detail for diagnosis
* Automatically retrying failed operations with exponential backoff
|===

===== Data Validation on Retrieval

[cols="1,4"]
|===
2+^|**Requirement {counter:req-id}**
|Identifier|/req/reliability/retrieval-validation
|Statement a|The system SHALL validate data consistency when retrieving stored data:

* Verify checksums for file-based retrievals
* Detect and report any data corruption to system administrators
* Provide alternative access to backup copies when primary data is corrupted
* Maintain audit logs of data integrity violations
|===

[NOTE]
====
Corruption detection relies on comparing stored checksums or hash values against
values recomputed at retrieval time. Alternative access to backup copies implies
that backup data is accessible independently of the primary data store; this may
be achieved through a secondary system, off-site backup, or disaster recovery
capability (see /rec/reliability/disaster-recovery).
====

[[rec_reliability_disaster_recovery]]
===== Disaster Recovery

[cols="1,4"]
|===
2+^|**Recommendation {counter:rec-id}**
|Identifier|/rec/reliability/disaster-recovery
|Statement a|The system SHOULD support disaster recovery capability including:

* A secondary system or equivalent standby capable of assuming operations following
  catastrophic failure of the primary system
* Documented switchover procedures, including manual activation steps and validation
  checks to confirm data integrity following switchover
* Replication or backup transfer to a geographically separate location
* Regular testing of disaster recovery procedures to verify recoverability
|===

[NOTE]
====
A disaster recovery capability (sometimes termed a cold standby or Disaster Recovery
Centre) provides resilience against catastrophic failure of the primary infrastructure
(fire, flood, hardware failure). It is distinct from high-availability clustering, which
addresses routine failures within the primary site. Switchover procedures should be
documented in system maintenance documentation as required by /req/maintainability/documentation.
====
